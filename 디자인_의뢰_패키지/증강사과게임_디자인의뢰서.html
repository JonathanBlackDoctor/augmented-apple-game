풀 비주얼 리디자인 의뢰

"합이 10이 되도록 숫자 사과를 드래그로 담는" 사과게임 코어 위에, 리그 오브 레전드 아레나식 증강 빌드와 실시간 1대1 대결을 얹은 웹 게임입니다. 게임 로직·기능은 거의 완성됐고, 이제 보이는 모든 것(UI·테마·배경·아이콘·모션·브랜딩)을 처음부터 다시 디자인하려 합니다.

이 문서의 전제 — 디자이너 전권 위임. 아래 "절대 제약"(§3) 몇 가지만 지켜주시면, 비주얼 컨셉·무드·컬러·스타일은 전적으로 디자이너의 제안을 따릅니다. 현재 화면 톤에 맞출 필요가 전혀 없습니다. 마음껏 갈아엎어 주세요.

01프로젝트 한눈에

바쁘면 이 페이지만 읽어도 됩니다.

한 줄 정의30초 안에 합이 10인 숫자 사과를 드래그로 최대한 많이 제거하고, 라운드 사이 증강을 쌓아 빌드를 만들어 1대1로 겨루는 캐주얼 경쟁 퍼즐 웹게임.
플랫폼웹 단일 코드베이스. 데스크톱 + 모바일(터치) 모두. 세로/가로. 320px 폭부터 대응.
기술React(메뉴·HUD·오버레이) + PixiJS(사과 보드 캔버스). 정적 배포(GitHub Pages) + Firebase 실시간 백엔드.
의뢰하는 것UI·테마·배경·게임 보드 아트·증강/티어 아이콘·모션·브랜딩까지 — 보이는 전부의 재설계 + 일관된 디자인 시스템.
창작 자유도전권 위임. §3의 절대 제약만 지키면 컨셉은 자유.
꼭 피할 것"제네릭 AI 룩"(특징 없는 보라-블루 그라데이션 템플릿). 사과게임 DNA(숫자 사과 격자 + 드래그 박스)는 유지.
가장 중요한 감각"손맛" — 드래그해서 사과가 펑 터지는 순간이 즉각적이고 만족스러워야 함(아래 4대 기둥 1순위).

왜 다시 디자인하나

현재 화면은 개발용으로 빠르게 만든 "따뜻한 다크 + 미니멀" 임시 룩입니다. 게임의 잠재력(중독적 코어 + 증강 빌드의 쾌감 + 실시간 경쟁)에 비해 비주얼 완성도가 못 미칩니다. 제품을 한 단계 끌어올릴 독창적이고 완성도 높은 비주얼 아이덴티티가 필요합니다.

02게임 이해 — 무엇을, 어떻게 노나

디자인 결정을 내리려면 게임이 어떻게 굴러가는지 알아야 합니다. 4개 레이어로 쌓여 있습니다.

① 코어 루프 — 사과게임 (Fruit Box)

  • 17열 × 10행 격자에 1~9 숫자가 적힌 사과가 빼곡히 깔립니다(총 170칸).
  • 마우스/손가락으로 직사각형을 드래그하면, 박스 안에 든 사과들의 숫자 합이 정확히 10일 때 모두 사라지고 점수를 얻습니다(사과 1개 = 1점).
  • 제한 시간 30초 동안 최대 점수를 노립니다. 빠른 판단 + 연속 제거의 리듬이 핵심.
  • 그래서 숫자 가독성과 제거 순간의 피드백이 디자인의 생명입니다.

② 증강 레이어 — 빌드 쌓기 (LoL 아레나식)

  • 라운드 사이마다 3개 중 1개 증강을 고릅니다(리롤 없음). 매치 내내 누적되어 "빌드"가 됩니다.
  • 티어가 오릅니다: 실버 (R1·R2) → 골드 (R3·R4) → 프리즘 (R5 고정). 티어가 높을수록 효과가 강하고 화려해야 합니다.
  • 6개 계열: 시간 · 콤보 · 보드 · 룰 · 하이리스크 · 견제. 각 계열은 고유한 시각 정체성(아이콘·색)이 필요합니다.
  • 증강이 쌓일수록 점수가 폭발하는 성장감이 느껴져야 합니다(4대 기둥 2순위).

③ 대결 레이어 — 실시간 1대1

  • 5라운드 × 30초. 라운드 점수 누적 + 라운드 승자 보너스 → 최종 1위 승.
  • 지금은 오프라인 AI 대결이 동작합니다(난이도 자동 조절). 온라인 사람 대 사람은 다음 단계.
  • MMR/ELO 랭크와 7단계 티어가 있습니다(§7). 승패에 따라 ΔMMR 연출이 들어갑니다.
  • 끊김 없는 동기화 + 신뢰 가능한 랭크 = 공정한 긴장감(4대 기둥 3순위).

④ 소셜 레이어 — 확산

  • 링크/카카오톡 공유, 친구 초대 보상, 데일리 시드 챌린지(모두 같은 보드로 점수 경쟁).
  • 공유 한 번이 새 유저·새 대결로 이어지는 바이럴 성장(4대 기둥 4순위) — 공유 카드/결과 화면이 자랑하고 싶게 생겨야 합니다.

디자인 우선순위 = 게임의 4대 기둥

1. 즉각적 손맛 (드래그·제거가 즉시 만족스럽게) › 2. 빌드의 쾌감 (증강이 쌓일수록 점수 폭발) › 3. 공정한 긴장감 (실시간 대결·랭크의 신뢰) › 4. 바이럴 성장 (공유가 확산으로). 비주얼·모션 결정은 이 순서대로 무게를 둬 주세요.

03절대 제약 & 자유 영역

아래 "지켜야 할 것"만 빼면 전부 자유입니다.

지켜야 할 것 (Hard constraints)

  • 사과게임 DNA 유지: 숫자가 적힌 사과 격자 + 드래그 박스 선택. 이 핵심 메타포는 바뀌지 않습니다(과일은 꼭 사과가 아니어도 되지만, "숫자 오브젝트 격자를 박스로 묶는" 구조는 유지).
  • 숫자 가독성 최우선: 모바일에서 셀 한 칸이 ~20px까지 작아집니다. 그 크기에서도 1~9가 즉시 읽혀야 합니다. 빽빽한 170칸이 시각적으로 피로하지 않아야 합니다.
  • 반응형: 320px(모바일 세로) → 데스크톱. 보드는 항상 화면에 꽉 맞게 들어가고, 터치 드래그와 마우스 드래그 모두 자연스러워야 합니다.
  • 성능: 보드는 드래그 중 60fps 미리보기 + 제거 파티클이 돕니다. 모바일을 무겁게 하는 과한 블러/그림자/필터 남발은 지양.
  • 30초 고텐션 가독성: 플레이 중 화면(보드·HUD)은 정보가 한눈에 들어와야 합니다. 장식이 판단을 방해하면 안 됩니다.
  • 한글 + 숫자: 모든 카피는 한국어. 폰트는 한글 전 음절 + 숫자 가독성 필수.

완전 자유 (Your call)

전체 컬러/무드 (다크·라이트·뭐든) 배경 컨셉·일러스트 사과/오브젝트 아트 스타일 타이포 선택 UI 컴포넌트 룩 증강·티어 아이콘 컨셉 모션·이펙트 언어 로고/워드마크 사운드 방향 제안(선택)

단 하나의 안티-패턴

특징 없는 "AI가 만든 것 같은" 룩 — 의미 없는 보라/청록 그라데이션, 글래스모피즘 떡칠, 어디서나 본 둥근 카드의 나열 — 은 피해 주세요. 이 게임만의 기억에 남는 정체성을 원합니다.

05게임 보드 아트 (가장 중요한 표면)

플레이어가 시간의 90%를 보는 화면입니다. 여기서 게임의 인상이 결정됩니다. 보드는 PixiJS 캔버스에 그려집니다(웹 DOM이 아님) — 디자이너는 벡터 스펙 또는 스프라이트 에셋으로 납품하면 개발이 캔버스에 반영합니다.

사과 — 기본

1~9 숫자. 가장 흔한 오브젝트 170개가 깔립니다. 작은 크기에서의 숫자 대비·가독성이 전부입니다.

사과 — 황금 (golden)

증강 "황금 사과"로 등장. 점수 2배. 일반과 한눈에 구분되되 과하지 않게.

사과 — 와일드 (wild)

아무 값으로 쓰이는 특수 사과. 시각적으로 "변환 가능"한 느낌.

빈 칸 (제거됨)

사과가 사라진 자리. 보드 바닥/패널이 보이는 상태.

선택 박스 — 진행 중

드래그 중이지만 아직 합이 10이 아님(중립 색). 합이 맞으면 즉시 "유효" 상태로 전환.

선택 박스 — 유효(=10)

합 10 달성. "지금 놓으면 제거!" 신호(현재는 초록). 가장 중요한 긍정 피드백.

제거 이펙트

사과가 터지는 파티클 버스트 + 짧은 사운드. 손맛의 핵심. 콤보 시 강해지면 좋음.

보드 배경/패널

격자 아래 깔리는 면. 사과 가독성을 해치지 않는 선에서 분위기를 만드는 곳.

크기 스펙 (가독성 기준)

셀(정사각형) 한 변은 화면에 맞춰 가변입니다. 모바일 최소 ~20px ~ 데스크톱 ~50px 범위를 가정하고, 숫자는 셀의 약 46% 크기로 들어갑니다. 두 극단 모두에서 또렷해야 합니다. 현재는 사과를 코드로 벡터 드로잉(원형 보디 + 상단 하이라이트 + 잎 줄기)하고 있으나, 스타일은 자유롭게 재정의해 주세요.

06증강 시스템 — 카드 & 아이콘 (전체 카탈로그)

증강은 카드로 제시되고, 각 증강에는 아이콘이 필요합니다. 6개 계열 각각의 시각 정체성 + 3개 티어의 위계가 핵심입니다. 아래는 전체 22종(현재 12종 구현, 나머지는 대결 단계에서 활성화).

10기술 제약 & 핸드오프 (구현 가능해야 합니다)

두 개의 렌더링 세계가 있습니다. 어디에 무엇을 납품하면 되는지.

① React / CSS 세계

홈·오버레이·HUD·메뉴·버튼·카드·배지·인풋·토스트. → CSS로 구현 가능한 스펙 + 디자인 토큰(CSS 변수)으로 납품. 컴포넌트 상태(hover/press/disabled/focus) 포함.

② PixiJS / 캔버스 세계

사과 보드·사과·선택 박스·파티클·견제 FX. → 벡터 드로잉 스펙(색·치수·비율) 또는 스프라이트 에셋(SVG, 그리고 래스터는 @1x/@2x/@3x)으로 납품. 현재는 코드 벡터 드로잉.

  • 반응형: 320px → 데스크톱. 보드 자동 맞춤(정사각 셀), 안전영역(노치) 대응(viewport-fit=cover).
  • 인터랙션: 마우스 드래그 + 터치 드래그, 텍스트 선택/탭 하이라이트 비활성, 전체화면 토글, 가로 모드 힌트.
  • 성능 예산: 드래그 미리보기·파티클 60fps. 모바일 GPU를 의식한 효과(과한 blur/filter 자제).
  • 폰트: 웹폰트 로딩 계획 포함(한글 서브셋 권장). 숫자 가독성 우선.
  • 다크/라이트: 한쪽만 해도 되고 둘 다 제안해도 됩니다(디자이너 판단). 단 토큰 구조로.

13협업 & 다음 단계

제안하는 진행 순서

  1. 킥오프 + 무드보드 — 방향 2~3안 제안(컨셉/컬러/타이포/사과 스타일).
  2. 파운데이션 + 핵심 화면 — 홈 · 게임 보드 · 증강 선택 · 결과 먼저 확정.
  3. 전체 시스템 — 컴포넌트·전 화면·전 상태·아이콘·티어.
  4. 모션 + 핸드오프 — 이펙트 스펙 + 토큰/에셋 납품.

킥오프 때 정하면 좋은 것 (디자이너 주도로)

  • 전체 톤: 다크/라이트, 캐주얼/프리미엄, 사실적/일러스트?
  • "사과"를 유지할지, 다른 모티프로 갈지(숫자 격자 구조는 유지).
  • 배경: 정적 일러스트 / 미세 모션 / 단색 — 어디까지?
  • 증강·티어 아이콘의 표현 양식(라인/면/3D/엠블럼).
